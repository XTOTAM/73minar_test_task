## Висновки

### Чи знайдено правильний контекст
- **q001–q003:** hybrid retrieval (FAISS + BM25) має знаходити відповідні секції за ключовими словами (відпустка, sick leave/лікарняний, payroll/indexation).
- **q004:** секція «Податкові строки» містить явну заборону називати точні дати.
- **q005:** секція індексації містить правило про відсутність числових даних для розрахунку.

### Чи відповідь не вигадує фактів
- Prompt + validator блокують вигадування дат (q004) та розрахунків (q005).
- Для q001 модель має посилатися на «6 місяців», а не підтверджувати право після 3 місяців.

### Чи правильно спрацював fallback
- **q004, q005:** обов'язковий fallback з `confidence: low`.
- **q001–q003:** fallback не потрібен, відповідь grounded у знайденому контексті.

### Слабкі місця
1. Змішані UA/EN секції — embedding може гірше працювати для коротких запитів.
2. Валідатор на regex-евристиках, без семантичної faithfulness-перевірки.
3. In-memory FAISS — не підходить для дуже великих KB без persistence.
4. Немає reranker між retrieval і LLM.

### Покращення для production (1–2 тижні)
1. Persistent vector store (Qdrant / PGVector) + webhook reindex при оновленні KB.
2. Cross-encoder reranker для top-10 → top-5.
3. Automated eval з golden answers і метриками context precision / faithfulness.
4. Post-generation fact-check: порівняння чисел у відповіді з numbers у sources.
5. OpenTelemetry + централізований trace store замість лише `traces.jsonl`.
6. LangGraph agent для multi-step запитів із tool-calling до payroll API.
